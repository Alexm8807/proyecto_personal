## 1. Resumen ejecutivo

Diseño y construcción de un portafolio profesional para posicionarse como **Product Systems Designer Estratégico** en empresas de Fintech scale-up, SaaS B2B complejo, data platforms, ERP e internal tools.

El portafolio reemplaza un Behance del 2020 que muestra entregas de UI cuando el valor real del perfil es sistémico y estratégico.

### Definición del perfil

Un perfil que opera en dos dimensiones simultáneas que raramente coexisten:

| Dimensión sistémica | Dimensión estratégica |
|---|---|
| Diseña reglas de comportamiento | Traduce sistemas complejos en experiencias comprensibles |
| Construye estructuras escalables | Trabaja el modelo mental del usuario |
| Garantiza consistencia de sistema | Domina UX writing y naming de features |
| *¿Cómo debería comportarse este sistema?* | *¿Cómo hacemos que el usuario lo entienda sin pensar demasiado?* |

Perfil adicional identificado: **Product Designer con pensamiento de PM** — piensa en reglas, se preocupa por cómo se explican y documenta decisiones de diseño. Perfil de alto impacto en equipos maduros (Shopify, Atlassian, Linear).

**El portafolio es en sí mismo el primer producto del nuevo perfil.**

---

## 2. Problema

### Problema de negocio (tuyo)
No existe un artefacto que comunique el perfil real. El Behance 2020 posiciona como UI Designer cuando el valor es sistémico y estratégico. Resultado: el perfil compite en el pool equivocado, con salarios y velocidad de contratación por debajo de su nivel real.

### Problema del usuario final (el reclutador)
Los hiring managers de productos B2B complejos buscan diseñadores que respondan esta pregunta con su trabajo:

> *¿Cómo hacemos que un sistema complejo se sienta simple para el usuario?*

Sin evidencia de ese pensamiento en el portafolio, el perfil no se diferencia del pool general de designers.

## 4. Audiencia objetivo

### Primaria — quien toma la decisión
**Hiring manager de producto B2B** en Fintech scale-up, SaaS complejo, ERP o data platform
- Busca: pensamiento sistémico + capacidad de hacer que la complejidad sea comprensible
- Pain principal: tienen ingenieros que construyen sistemas complejos pero diseñadores que no logran hacerlos adoptables

### Secundaria — quien filtra primero
**Tech recruiter** especializado en producto B2B
- Busca: señales rápidas de seniority sistémico + stack relevante + impacto medible
- Comportamiento: 20–30 segundos en el home, luego abre un caso, luego el about

### Descartado explícitamente
- Startups early-stage

### Empresas de referencia donde este perfil encaja
Airtable · Notion · Figma · Atlassian · Shopify · Linear · Vercel · y equivalentes en Fintech / ERP

## 5. Propuesta de valor

> *"Diseño sistemas complejos que los usuarios entienden. Ahora también los construyo."*

**Capa 1 — Identidad:** Product Systems Designer Estratégico  
**Capa 2 — Diferenciador:** no solo diseña la lógica del sistema, traduce esa lógica en experiencias que reducen carga cognitiva  
**Capa 3 — Multiplicador:** las AI tools (Stitch, Lovable, Figma Make, Claude Code) permiten ejecutar ese pensamiento directamente, sin depender del ciclo de handoff

## 6. Alcance del producto

### Fase 1 — Portafolio puente (prioridad alta, lanzar primero)

**Páginas:**
- **Home / Hero** — narrativa central clara en < 3 segundos de lectura + señal técnica inmediata
- **Casos (×2)** — casos UX existentes reescritos con formato STAR sistémico y métricas
- **About** — punto de vista propio, flujo de trabajo real, stack técnico
- **Contacto**

**Criterio de done de Fase 1:**
- [ ] Construido con herramientas del stack declarado
- [ ] Dominio propio activo con HTTPS
- [ ] Lighthouse Performance ≥ 90
- [ ] Accesibilidad WCAG 2.1 AA
- [ ] 2 casos con problema sistémico, decisiones documentadas y métrica de impacto
- [ ] Mobile responsive

### Fase 2 — Portafolio completo (cuando existan proyectos builder)

**Adiciones:**
- Builder hero case con link a producto live
- Caso de design system con decisiones estratégicas visibles (naming, comportamiento, descartados)
- Sección "Cómo trabajo" — flujo documentado con AI tools, no lista de logos

### Fuera de alcance (por ahora)
- Blog o sección editorial
- Versión multiidioma
- Animaciones complejas o Three.js
- CMS para gestión de contenido

## 8. Requisitos no funcionales

| Requisito | Especificación | Justificación |
|---|---|---|
| Performance | Lighthouse ≥ 90 | Los hiring managers de producto revisan esto |
| Accesibilidad | WCAG 2.1 AA mínimo | Señal clave para audiencia Fintech y ERP; coherente con el perfil sistémico |
| Tiempo de carga | < 2 segundos | Estándar de la audiencia objetivo |
| Tipografía | Body ≥ 16px, contraste AAA en textos principales | El writing del portafolio es evidencia del UX writing skill |
| Dominio | Propio, HTTPS | Sin Behance, sin soluciones genéricas |
| Coherencia técnica | El stack del sitio debe coincidir con el stack declarado | Disonancia técnica = señal negativa inmediata |

## 9. Estructura de contenido

### Home
- Headline: narrativa central (quién eres en el contexto B2B complejo)
- Sub-headline: el diferenciador estratégico (sistemas complejos → experiencias comprensibles)
- Señal técnica: stack o producto live visible
- Acceso directo a casos
- CTA de contacto

### Caso (estructura STAR sistémico)
1. Contexto de negocio B2B y complejidad del dominio
2. Problema sistémico (reglas, datos, operaciones que generaban confusión)
3. Mi rol exacto
4. Decisiones de sistema (comportamiento, estructura, consistencia)
5. Decisiones de comunicación (naming, carga cognitiva, progressive disclosure)
6. Resultado con métrica (adopción, soporte, conversión, onboarding)
7. Lo que haría diferente

### About
- Punto de vista propio sobre diseño de sistemas complejos
- Flujo de trabajo real (del problema al sistema comprensible)
- Cómo las AI tools aceleran ese proceso
- Stack técnico actual

## 11. Riesgos y mitigaciones

| Riesgo | Probabilidad | Mitigación |
|---|---|---|
| Posicionarse demasiado amplio y perder el foco B2B | Media | El headline debe nombrar el contexto (sistemas complejos, B2B) explícitamente |
| Casos sin evidencia de decisiones sistémicas | Alta | Reescribir casos existentes poniendo foco en las decisiones de sistema y comunicación, no en los entregables visuales |
| Subestimar el perfil al llamarse "UI Designer" o "Builder Designer" | Alta | El naming del rol en el portafolio debe reflejar el nivel real: Product Systems Designer o Staff Product Designer |
| Lanzar tarde por esperar proyectos builder perfectos | Media | Fase 1 no los requiere. Lanzar primero, iterar después |
| El writing del portafolio contradice el UX writing skill declarado | Baja | El texto de casos y about debe ser claro, sin jerga, sin saturación — evidencia directa del perfil |
